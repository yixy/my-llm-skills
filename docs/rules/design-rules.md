# 设计规范

本文档按触发场景组织。操作前查阅对应章节。

> 标记说明：【强制】= 必须遵守，违反将导致缺陷；【推荐】= 应尽量遵守，特殊情况下可豁免

---

## 建表时适用

编写 DDL、Flyway migration、或者创建新表时遵守：

- 【强制】不使用某个数据库专有的 SQL 语法，只使用标准 SQL 语法
- 【强制】表名命名以应用标识 `eaam_` 开头
- 【强制】考虑软删除和审计字段：增加 `is_deleted`、`created_at`、`updated_at` 等字段
- 【强制】每个表必须有主键；必须建立业务主键并创建唯一索引；必须包含表及字段级别 comment
- 【强制】活跃表中禁止使用 blob、text、varchar(>500) 等大字段
- 【强制】IP 相关字段设计时需支持 IPv4 和 IPv6 协议
- 【强制】数据库名、表名、字段名全部使用英文小写字母，不超过 32 字符，只包含字母、数字和下划线
- 【强制】使用有意义的英文单词命名表和字段，避免使用数据库保留关键字
- 【强制】varchar(2000)、text、blob 等大字段需与主键分拆到单独表存放
- 【强制】多个表中相同属性的字段类型、属性和长度完全一致
- 【强制】分片键类型为 int、bigint、smallint、char、varchar；分片键值统一使用小写
- 【强制】单表索引不超过 5 个；避免冗余索引；联合索引区分度高的字段在前
- 【强制】使用 DATE、TIME、TIMESTAMP 等适当的数据类型表示日期和时间
- 【推荐】对于大字段（如 CLOB、BLOB）的使用需谨慎，避免滥用
- 【推荐】分区表的索引原则上必须建为分区索引，对分区表的查询需要带分区键，索引建为分区索引
- 【推荐】表设计需考虑适当的反范式设计，在保证数据一致性的前提下，对频繁查询但不频繁修改的字段进行适当冗余
- 【推荐】表字段更新时间字段建议添加索引（`update_time` 字段添加索引，或作为组合索引的最左边字段）

## API 设计时适用

设计或修改 Controller / REST API 接口时遵守：

- 【强制】接口设计需考虑安全性要求，包含输入验证、输出编码、访问控制（认证、授权、加密）等安全措施
- 【强制】接口设计需考虑版本兼容性，包含版本信息，支持向后兼容
- 【强制】接口设计需考虑错误处理机制，包含完善的错误处理机制，返回明确的错误信息
- 【推荐】接口设计需考虑性能优化，合理使用分页、缓存等措施
- 【推荐】接口设计需考虑限流机制，防止恶意请求或流量冲击
- 【推荐】接口设计包含完整的日志记录，便于问题排查

---

## 模块设计时适用

新增模块或拆分现有模块时遵守：

- 【强制】模块设计需遵循高内聚低耦合原则，模块内部功能紧密相关，模块间依赖关系清晰
- 【强制】模块设计需考虑可重用性，避免重复开发
- 【强制】模块设计需考虑可扩展性，便于功能扩展
- 【强制】模块设计需清晰，便于维护和修改

---

## 程序逻辑编写时适用

编写 Service 层、业务逻辑代码时遵守：

- 【强制】程序逻辑需遵循输入验证规范，对通讯协议中从客户端传过来的一切变量进行严格的安全性检查
- 【强制】程序逻辑需对共享资源进行适当的并发控制，避免数据不一致
- 【强制】程序逻辑需合理使用事务，确保数据一致性
- 【强制】程序逻辑需考虑异常处理，对可能出现的异常进行适当处理，确保程序健壮性
- 【强制】程序逻辑需考虑性能优化，避免使用性能较差的操作（如循环中频繁创建对象、过度使用同步等）
- 【强制】程序逻辑需考虑资源释放，及时释放数据库连接、文件句柄等资源，避免资源泄漏
- 【推荐】程序逻辑需考虑可维护性，程序逻辑清晰，代码结构合理，便于维护和扩展
- 【推荐】程序逻辑需考虑可测试性，程序逻辑模块化，便于单元测试和集成测试
- 【推荐】程序逻辑需考虑安全性，避免安全漏洞（如 SQL 注入、跨站脚本等）
- 【推荐】程序逻辑需考虑容错性，包含容错机制，提高系统健壮性

## JSON 报文时适用

定义 DTO、Request/Response Body、序列化相关代码时遵守：

- 【强制】JSON 对象中的成员名称（键）必须保持唯一性约束
- 【强制】程序逻辑不得依赖 JSON 对象中成员的物理出现顺序
- 【强制】JSON 报文中所有键名（Key）和字符串类型的值（Value）必须使用半角双引号（"）包裹，严禁使用单引号（'）或无引号形式
- 【强制】JSON 报文必须严格执行字符转义规则：对于双引号（"）、反斜线（\）以及所有 ASCII 控制字符（U+0000 至 U+001F）进行合规转义，以防止报文截断或解析异常
- 【推荐】JSON 报文中对于数值长度超过 15 位的字段（如大额交易金额、长整型 ID 等），建议以字符串（String）形式传递
- 【推荐】JSON 报文中键名建议仅由半角英文字母、数字、下划线（_）组成，且必须以字母开头
- 【推荐】JSON 报文中键名建议遵循 lowerCamelCase（小驼峰式）命名法
- 【推荐】JSON 报文中避免仅通过键名大小写区分字段
- 【推荐】JSON 报文中单个键名的长度不宜超过 64 个字符
- 【推荐】在定义 JSON 报文的顶级键名时，宜避开各主流编程语言的关键字（如 class、type、id、default、function 等）

---

## 批处理设计时适用

设计批处理任务、定时任务时遵守：

- 【强制】批处理需要支持断点续跑
- 【强制】批处理设计要防止重复入账
- 【强制】大数量处理时需要支持多任务切片处理，提升处理效率
- 【强制】批量语句中使用绑定变量，提高性能和安全性
- 【强制】批量中临时表的统计信息收集需人工干预
- 【推荐】批处理设计需考虑日志记录
- 【推荐】批处理设计需考虑监控机制
- 【推荐】批处理设计需考虑告警机制
